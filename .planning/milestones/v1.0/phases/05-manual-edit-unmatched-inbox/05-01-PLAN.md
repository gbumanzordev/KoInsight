<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser -> Express | Untrusted JSON body crosses at `PATCH /api/books/:bookId/metadata`; route validates via Zod before any DB call. |
| Express -> SQLite (Knex) | Parameterized queries only; no raw SQL string interpolation of body values. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Tampering | PATCH /:bookId/metadata body | mitigate | `metadataPatchSchema.strict()` rejects unknown keys at the route boundary; only the enumerated four fields reach Knex. No `id`, `md5`, `enrichment_status`, `*_source` column is in the schema, so mass assignment is structurally impossible. |
| T-05-02 | Tampering | SQL injection via `:bookId` | mitigate | `getBookById` middleware calls `Number(req.params.bookId)` and `BooksRepository.getById(numericId)`; Knex parameterizes. No string concat. |
| T-05-03 | Tampering | SQL injection via authors/genres strings | mitigate | All `.whereIn()` / `.where()` / `.update({...})` / `.insert({...})` calls are parameterized by Knex. No `db.raw` with interpolated values in `applyManualEdit`. |
| T-05-04 | DoS | Oversized `authors` array | mitigate | `authorEditSchema` array capped at `.max(50)`; genres capped at `.max(50)`. Express `json({ limit: '50mb' })` is the outer cap. |
| T-05-05 | Repudiation | Manual edit audit trail | accept | Single-user self-host deployment model per PROJECT.md; no audit log requirement this milestone. `*_source='manual'` is the only audit signal (insufficient for legal audit but sufficient for correctness). |
| T-05-06 | Elevation of Privilege | IDOR via numeric bookId | accept | Single-user deployment; no multi-user auth model exists. Any request reaching the server is already privileged. Explicitly out of scope per RESEARCH.md security section. |
| T-05-07 | Information Disclosure | Zod error leaking internals | mitigate | `parsed.error.flatten()` returns only fieldErrors/formErrors keyed by schema field names; no stack traces, no DB details. |

**ASVS L1 coverage:** V5 Input Validation satisfied via Zod schema at boundary with `.strict()`. V2/V3/V4 N/A (no auth model). V6 N/A (no new crypto).
</threat_model>
