  <behavior>
    `enrichmentService.enqueue(bookMd5: string): Promise<void>`:
    - Validates `bookMd5` with Zod `/^[a-f0-9]{32}$/i`. On parse failure, `console.warn('enrichment enqueue: invalid md5', { bookMd5 })` and return (D-09 log-and-swallow; no throw).
    - D-07 predicate: SELECT `enrichment_status` from `book WHERE md5 = ?`. If row does not exist, or `enrichment_status` IS `'enriched' | 'failed' | 'skipped' | 'running'`, do NOT enqueue. Enqueue only when `enrichment_status` IS NULL OR `'pending'`.
    - D-08 dedup: INSERT INTO enrichment_job (book_md5, status) VALUES (?, 'pending') ON CONFLICT DO NOTHING (via knex builder `.onConflict().ignore()` or `knex.raw` if builder chokes on the partial index, see Pattern note below).
    - D-09: any DB error (connection, IO, unexpected) is caught and `console.warn('enrichment enqueue failed', { bookMd5, phase: 'enqueue', err: String(err) })`. Never throw.

    Test coverage (using real `:memory:` SQLite via test-setup):
    - Enqueue for a book with `enrichment_status = NULL`: one row inserted in enrichment_job with status='pending'.
    - Enqueue for a book with `enrichment_status = 'pending'`: one row inserted.
    - Enqueue for a book with `enrichment_status = 'enriched'`: no row inserted.
    - Enqueue for a book with `enrichment_status = 'failed'`: no row inserted.
    - Enqueue for a book with `enrichment_status = 'skipped'`: no row inserted.
    - Enqueue for a non-existent md5: no row inserted, no throw.
    - Invalid md5 format ("not-an-md5"): no row inserted, no throw, console.warn called (spy).
    - Concurrent enqueue (two Promises for the same md5): exactly ONE open row in enrichment_job (D-08 invariant).
    - Enqueue for md5 A when md5 A already has a row with status='running': no new row inserted (partial UNIQUE catches it).
    - Enqueue for md5 A when A has a row with status='failed' (closed): a new pending row CAN be inserted (partial UNIQUE only applies to open states). This case is the edge that matters for Phase 5's re-enrich endpoint but is also the sync-path behavior for re-synced books that previously failed, note: D-07 actually blocks this case because book.enrichment_status = 'failed' means no enqueue. Test BOTH layers: D-07 predicate blocks, AND the DB-layer partial UNIQUE does NOT block.

     ```typescript
     import { z } from 'zod';
     import { db } from '../knex';

     const Md5Schema = z.string().regex(/^[a-f0-9]{32}$/i);

     export const enrichmentService = {
       async enqueue(bookMd5: string): Promise<void> {
         const parsed = Md5Schema.safeParse(bookMd5);
         if (!parsed.success) {
           console.warn('enrichment enqueue: invalid md5', { bookMd5 });
           return;
         }
         try {
           const book = await db('book').select('enrichment_status').where({ md5: bookMd5 }).first();
           if (!book) return;
           const status = book.enrichment_status as string | null;
           if (status !== null && status !== 'pending') return;

           await db('enrichment_job').insert({ book_md5: bookMd5, status: 'pending' }).onConflict().ignore();
         } catch (err) {
           console.warn('enrichment enqueue failed', { bookMd5, phase: 'enqueue', err: String(err) });
         }
       },
     };

     export const { enqueue } = enrichmentService;
     ```

     Note on `.onConflict().ignore()` with SQLite partial UNIQUE: the Knex 3.1.0 builder supports no-arg `.onConflict().ignore()` which lowers to `OR IGNORE` (upsert alternative). If a test shows this is insufficient against the partial index specifically, fall back to `knex.raw('INSERT OR IGNORE INTO enrichment_job (book_md5, status) VALUES (?, ?)', [bookMd5, 'pending'])`. `INSERT OR IGNORE` in SQLite treats any UNIQUE constraint violation (partial or full) as a no-op, which is the desired behavior per D-08. Plan executor confirms which form works by running the concurrent-enqueue test.
