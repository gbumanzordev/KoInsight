<objective>
Migration 4 of 4 (the ONLY data-only migration in Phase 1, per D-02): iterate over existing `book` rows whose `authors` string is non-empty, parse using the pure `parseAuthors` helper from Plan 01, dedup authors via the normalized key from D-09, insert (or look up) author rows, and create the corresponding `book_author` junction rows with contiguous `position` values starting at 0.

Purpose: SCHEMA-08. This is the bridge from the legacy denormalized `book.authors` text column to the normalized author entity model. After this migration, every legacy book has its authors materialized as entities; new books written by KOReader sync will continue to populate `book.authors` (SCHEMA-03 invariant) and Phase 4's enrichment service will be responsible for populating `book_author` for new books going forward.
Output: One migration file that imports the pure parser, runs in a single Knex transaction, performs lookup-before-insert on `author`, inserts `book_author` rows with contiguous positions.
</objective>

    ```typescript
    import type { Knex } from 'knex';
    import { parseAuthors } from './helpers/parse-authors';

    // D-09: app-layer dedup key. No schema column for this; UNIQUE(name) is the schema backstop (D-08).
    function dedupKey(name: string): string {
      return name.trim().replace(/\s+/g, ' ').toLowerCase();
    }

    export async function up(knex: Knex): Promise<void> {
      await knex.transaction(async (trx) => {
        // SCHEMA-08: source data is existing book.authors strings only. No external data, no network.
        const books: Array<{ md5: string; authors: string }> = await trx('book')
          .select('md5', 'authors')
          .whereNotNull('authors')
          .andWhere('authors', '!=', '');

        // Cache: dedupKey -> author.id, populated lazily so we make at most one INSERT per unique author.
        const authorIdByKey = new Map<string, number>();

        for (const book of books) {
          const parsed = parseAuthors(book.authors);
          if (parsed.length === 0) continue;

          for (const { name, position } of parsed) {
            const key = dedupKey(name);

            let authorId = authorIdByKey.get(key);

            if (authorId === undefined) {
              // Look up by exact name first (UNIQUE(name) at schema level — D-08).
              // Because dedupKey is case-insensitive but UNIQUE(name) is case-sensitive,
              // we search by the dedupKey via LOWER(TRIM(...)) to honor D-09.
              const existing = await trx('author')
                .select('id')
                .whereRaw("LOWER(TRIM(REPLACE(REPLACE(name, '  ', ' '), '  ', ' '))) = ?", [key])
                .first();

              if (existing) {
                authorId = existing.id;
              } else {
                // Insert. UNIQUE(name) is the backstop: if a concurrent migration somehow
                // inserted the same canonical name, we'd get a UNIQUE error — but migrations
                // run serially so that path is not realistic.
                const [inserted] = await trx('author')
                  .insert({
                    name, // D-08: stored verbatim (case + punctuation as parsed)
                    nationality_source: null, // D-14
                    bio: null,
                  })
                  .returning('id');
                authorId = typeof inserted === 'object' ? inserted.id : inserted;
              }

              authorIdByKey.set(key, authorId as number);
            }

            // Insert junction row. role defaults to 'author' (set in Plan 03's schema).
            // unique(book_md5, position) and unique(book_md5, author_id) are the schema guards.
            await trx('book_author').insert({
              book_md5: book.md5,
              author_id: authorId,
              position,
              role: 'author',
            });
          }
        }
      });
    }

    export async function down(knex: Knex): Promise<void> {
      // Data-only migration. Rollback drops only the data this migration inserted.
      // Safe approach: truncate book_author and author. This is acceptable because
      // Phase 1 is the first phase to populate these tables — there is no upstream data
      // we would be destroying. (Phase 4 enrichment also writes here, but if you are
      // rolling back Phase 1, you have already rolled back later phases first.)
      await knex('book_author').del();
      await knex('author').del();
    }
    ```

    Notes on choices:
    - Single `knex.transaction` wraps the entire backfill (per CONTEXT §specifics). Mid-failure leaves the DB unchanged.
    - The `whereNotNull('authors').andWhere('authors', '!=', '')` filter avoids invoking the parser on empty/null strings.
    - The `Map<dedupKey, authorId>` cache means at most one DB lookup per unique author across the whole backfill — orders of magnitude faster than re-querying per book.
    - The lookup uses `LOWER(TRIM(REPLACE(REPLACE(name, '  ', ' '), '  ', ' ')))` to approximate D-09's `name.trim().replace(/\s+/g, ' ').toLowerCase()`. SQLite has no native regex; the double-REPLACE collapses runs of up to 4 spaces, which is enough for any realistic display name. (If exact regex parity becomes critical later, Phase 4 can refine the cache.)
    - D-10: NO Unicode folding. We do not call NOCASE / NOACCENT collations.
    - D-12: NO duplicate-author merge by OL key in this migration; Phase 1 has only names, not OL keys. Phase 4 owns merge.
    - The `down()` truncates author + book_author. This is destructive but acceptable because Phase 1 is the bottom of the dependency stack — rolling Phase 1 down implies rolling all downstream data too.
