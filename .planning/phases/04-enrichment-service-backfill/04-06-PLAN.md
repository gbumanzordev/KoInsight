  Create one test file with five describe blocks.

  **Common harness (shared beforeEach across all describes):**
  - Use `vi.useFakeTimers({ shouldAdvanceTime: false })` at `beforeEach`; `vi.useRealTimers()` at `afterEach`.
  - `vi.stubGlobal('fetch', mockFetch)` where `mockFetch` routes by URL substring to the fixture JSONs (per PATTERNS section):
    ```typescript
    const mockFetch = vi.fn().mockImplementation(async (url: string) => {
      if (url.includes('/search.json')) return jsonResponse(searchFixture);
      if (url.includes('/books/')) return jsonResponse(editionFixture);
      if (url.includes('/works/')) return jsonResponse(workFixture);
      if (url.includes('/authors/')) return jsonResponse(authorFixture);
      if (url.includes('wikidata.org')) return jsonResponse(wikidataFixture);
      throw new Error('unexpected fetch url: ' + url);
    });
    ```
  - Seed the `genre` table with Phase 2's CANONICAL_GENRES at the start of each scenario. Use the same seed helper Plan 04 tests used.
  - Helper `async function runTicks(n: number)`: advance fake timer by `n * ENRICHMENT_POLL_INTERVAL_MS`, each step followed by `await vi.runAllTimersAsync()` to drain the microtask queue per PATTERNS Pitfall 4.
  - Helper `async function startWorker()`: calls `startEnrichmentWorker(db)`, awaits a short delay so the crash-recovery sweep completes, returns the worker handle.

  **Describe 1: SC-1 "sync -> enqueue -> tick -> enriched"**
  - Seed `book` row with md5 = Ender's Game hash (any 32-char hex), `title = "Ender's Game"`, `authors = "Orson Scott Card"`, `enrichment_status = NULL`.
  - Call `await enrichmentService.enqueue(md5)`. Assert `enrichment_job` has one row with status='pending'.
  - Start worker. Run one tick.
  - Assert: `book.enrichment_status = 'enriched'`, `book.openlibrary_work_key` set, `book.publication_year` set (per fixture), `book_author` has at least one row, `book_genre` has at least one row (mapped from fixture subjects), `enrichment_job.status = 'succeeded'`.
  - Call `worker.stop()` at the end.

  **Describe 2: SC-3 "idempotent: two runs produce identical state"**
  - Run scenario 1 to completion.
  - Snapshot `book`, `book_author` (ordered), `book_genre` (sorted by genre_id). Exclude timestamp columns (created_at, updated_at) from the snapshot.
  - Re-enqueue the same md5 (requires temporarily setting book.enrichment_status = 'pending' to bypass D-07 — document this as the test simulating a Phase 5 explicit re-enrich path, since Phase 4 by itself has no re-enqueue path for already-enriched books).
  - Run another tick.
  - Snapshot again. Deep-equal the two snapshots. `book_author` row ordering must match; `book_genre` set must match.

  **Describe 3: SC-4 "manual-wins on genres survives re-enrichment"**
  - Seed `book` with `genres_source = 'manual'` and one manually-inserted `book_genre` row (e.g., genre 'Fantasy').
  - Stub fetch with a version of `workFixture` whose subjects map to a DIFFERENT canonical genre (e.g., 'Science Fiction').
  - Enqueue + tick (set enrichment_status to 'pending' to allow enqueue).
  - Assert: `book_genre` is STILL the single Fantasy row; `book.genres_source` still 'manual'.
  - Also assert `book.enrichment_status = 'enriched'` (the apply still runs; just not on the genre column).

  **Describe 4: SC-5 part 1 "crash-recovery sweep on restart"**
  - Seed `enrichment_job` directly: `{ book_md5, status: 'running', attempts: 1 }`. (Simulates the post-crash state.)
  - Call `startEnrichmentWorker(db)`.
  - Immediately (before any tick) query the row. Assert `status = 'pending'`.
  - Optionally run a tick; assert the worker then processes normally and transitions to 'succeeded'.

  **Describe 5: SC-5 parts 2+3 + ENRICH-07 "no-match -> book.enrichment_status='failed'"**
  - Seed book with title that will not match fixture's search docs (e.g., override mockFetch to return `{ docs: [] }` for this scenario).
  - Enqueue + tick.
  - Assert: `book.enrichment_status = 'failed'`, `enrichment_job.status = 'failed'`, `enrichment_job.last_error` contains 'no-match'.

  Max-attempts ceiling is tested thoroughly in Plan 05's worker test; do NOT duplicate here (SC-5 part 2 "max-attempts leaves failed with last_error" is already locked).

  Follow Phase 3 integration test file structure exactly (import shape, beforeEach, describe blocks, jsonResponse helper). Aim for ~400 lines.
